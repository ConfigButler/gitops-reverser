# E2E Test Design: Aggregated API Server (Extension API Server)

> Last updated: April 2026

## Motivation

GitOps Reverser currently has e2e coverage for built-in resources (ConfigMaps, Secrets, Deployments,
etc.) and CRD-backed resources (IceCreamOrder). However, Kubernetes supports a third category of
resources: those served by **aggregated API servers** (extension API servers), registered via
`APIService` objects.

These resources participate in the Kubernetes API surface differently from CRDs:

- They are served by a separate process (not etcd-backed through the core API server)
- The core API server proxies requests to the extension server
- Audit events are generated by the core API server's proxy layer, not the extension server
- Watch streams are proxied through the aggregation layer
- Resource names served by aggregated APIs are **not required to follow DNS label rules** — they can
  contain colons, dots, and other characters that CRD names cannot (e.g., `pods` in
  `custom.metrics.k8s.io` can have names like `pods/cpu_usage_rate`)
- Status subresources may behave differently depending on the extension server implementation

We need to verify that the full gitops-reverser pipeline — audit ingestion, watch/informer
reconcile, sanitization, file path generation, and git commit — works correctly with aggregated API
resources.

### What the existing docs already say

[kubernetes-api-discovery.md](kubernetes-api-discovery.md) already states that aggregated API
resources are theoretically supported: "yes, you can watch resources served through `APIService`;
CRD is not required." However, this has **never been tested end-to-end**. This design closes that
gap.

---

## Scope

This test suite validates:

1. **Audit webhook behavior** — do audit events for aggregated API resources arrive with the same
   structure (verb, objectRef, responseObject) as CRD/built-in resources?
2. **Watch/informer behavior** — can informers list+watch aggregated API resources for
   snapshot/reconcile?
3. **Status subresource handling** — does a status update on an aggregated resource produce the
   expected audit event and git commit (or correctly get filtered)?
4. **Non-DNS-valid resource names** — do resource names that contain colons, underscores, or other
   non-DNS characters survive the full pipeline (rule matching → sanitize → file path → git)?
5. **API discovery integration** — does `FilterDiscoverableGVRs()` correctly discover and accept
   aggregated API resources?
6. **Reconcile/snapshot** — does an initial snapshot correctly capture existing aggregated API
   resources?

Out of scope:
- Extension API server high availability or failure modes
- Performance comparison between CRD and aggregated API watches
- mTLS between the core API server and extension server (handled by Kubernetes, not our code)

## Test Infrastructure: Which Extension API Server to Use

### Option A: `kubernetes/sample-apiserver` (Recommended)

The Kubernetes project maintains
[sample-apiserver](https://github.com/kubernetes/sample-apiserver) — a minimal extension API
server that serves a resource called `Flunder` in the `wardle.example.com` API group. This is
the canonical example used by Kubernetes itself for testing aggregated APIs.

**Advantages:**
- Official Kubernetes project, actively maintained alongside k8s releases
- Minimal footprint: single binary, single CRD-like resource (`Flunder`), simple schema
- Has a `status` subresource
- Supports `list`, `watch`, `get`, `create`, `update`, `patch`, `delete`
- Used extensively in Kubernetes' own e2e and integration tests
- Pre-built container images are available (or trivially buildable)
- Supports non-DNS-valid names in resource instances (because the extension server controls
  its own validation)
- Well-documented setup including `APIService` registration and TLS

**Resource it serves:**
```
Group:    wardle.example.com
Version:  v1beta1
Resource: flunders (namespaced)
Kind:     Flunder
```

Also serves `fischers` (cluster-scoped), which is useful for testing `ClusterWatchRule`.

**Disadvantages:**
- Needs TLS certificates for the aggregation layer (cert-manager already available in our e2e
  cluster)
- Adds a pod to the test cluster

### Option B: `metrics-server` (Not recommended)

The metrics server is an aggregated API server (`metrics.k8s.io`), but its resources (`nodes`,
`pods`) are read-only (no create/update/delete), making it unsuitable for testing the full
mutating audit → git pipeline.

### Option C: Custom minimal extension server (Not recommended)

Building our own would give full control but would duplicate the sample-apiserver's purpose,
add maintenance burden, and delay the test implementation.

### Option D: `apiregistration.k8s.io` mock (Not recommended)

Registering an `APIService` that points to a mock HTTP server. Too fragile — would need to
implement the full Kubernetes API contract (list, watch, version negotiation, etc.).

**Decision: Option A (`sample-apiserver`)** — it is purpose-built for this exact use case, is
maintained by the Kubernetes project, and covers all the resource types and operations we need.

## Setup: Deploying `sample-apiserver` in k3d

### Deployment artifacts

Create a new directory for the aggregated API test infrastructure:

```
test/e2e/setup/sample-apiserver/
├── kustomization.yaml
├── namespace.yaml
├── deployment.yaml
├── service.yaml
├── apiservice.yaml
├── rbac.yaml

```

### Key resources

**Namespace:**
```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: wardle-system
```

**Deployment** (sample-apiserver binary):
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sample-apiserver
  namespace: wardle-system
spec:
  replicas: 1
  selector:
    matchLabels:
      app: sample-apiserver
  template:
    spec:
      containers:
      - name: apiserver
        image: registry.k8s.io/e2e-test-images/sample-apiserver:1.32.1
        args:
        - --tls-cert-file=/etc/certs/tls.crt
        - --tls-private-key-file=/etc/certs/tls.key
        - --etcd-servers=       # empty = in-memory storage (sufficient for e2e)
        - --feature-gates=APIPriorityAndFairness=false
        volumeMounts:
        - name: certs
          mountPath: /etc/certs
          readOnly: true
      volumes:
      - name: certs
        secret:
          secretName: sample-apiserver-tls
```

**Service:**
```yaml
apiVersion: v1
kind: Service
metadata:
  name: sample-apiserver
  namespace: wardle-system
spec:
  ports:
  - port: 443
    targetPort: 443
  selector:
    app: sample-apiserver
```

**cert-manager Certificate:**
```yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: sample-apiserver-tls
  namespace: wardle-system
spec:
  secretName: sample-apiserver-tls
  issuerRef:
    name: selfsigned-issuer       # reuse existing e2e cluster issuer
    kind: ClusterIssuer
  dnsNames:
  - sample-apiserver.wardle-system.svc
  - sample-apiserver.wardle-system.svc.cluster.local
```

**APIService** (registers the aggregated API):
```yaml
apiVersion: apiregistration.k8s.io/v1
kind: APIService
metadata:
  name: v1beta1.wardle.example.com
spec:
  group: wardle.example.com
  version: v1beta1
  service:
    name: sample-apiserver
    namespace: wardle-system
  groupPriorityMinimum: 1000
  versionPriority: 15
  caBundle: <injected-by-cert-manager-or-script>
```

**RBAC:**
```yaml
# The sample-apiserver needs auth-delegator permissions
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: sample-apiserver:system:auth-delegator
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: system:auth-delegator
subjects:
- kind: ServiceAccount
  name: default
  namespace: wardle-system
```

Plus: gitops-reverser's ClusterRole needs `get/list/watch` on `flunders.wardle.example.com` and
`fischers.wardle.example.com`.

### Integration into Taskfile

Add a new stamp-based task:

```yaml
_sample-apiserver-ready:
  method: timestamp
  sources:
    - test/e2e/setup/sample-apiserver/**/*
  generates:
    - '{{.CS}}/sample-apiserver.ready'
  status:
    - test -f "{{.CS}}/sample-apiserver.ready"
  cmds:
    - task: _services-ready
    - |
      kubectl --context "{{.CTX}}" apply -k test/e2e/setup/sample-apiserver
      kubectl --context "{{.CTX}}" -n wardle-system wait deployment/sample-apiserver \
        --for=condition=Available --timeout=120s
      # Wait for APIService to become available
      kubectl --context "{{.CTX}}" wait apiservice/v1beta1.wardle.example.com \
        --for=condition=Available --timeout=120s
      # Verify the resource is discoverable
      kubectl --context "{{.CTX}}" api-resources | grep -q flunders
      touch "{{.CS}}/sample-apiserver.ready"

test-e2e-aggregated-api:
  desc: Run aggregated API server e2e scenarios
  deps:
    - _sample-apiserver-ready
  cmds:
    - |
      export CTX="{{.CTX}}"
      export INSTALL_MODE="{{.INSTALL_MODE}}"
      export NAMESPACE="{{.NAMESPACE}}"
      export E2E_AGE_KEY_FILE="{{.CS}}/age-key.txt"
      go test -timeout "{{.E2E_GO_TEST_TIMEOUT}}" ./test/e2e/ -v -ginkgo.v \
        -ginkgo.label-filter=aggregated-api
```

### Audit policy update

The existing audit policy at `test/e2e/cluster/audit/policy.yaml` already has a catch-all rule at
the bottom:

```yaml
- level: RequestResponse
  omitManagedFields: true
  verbs: [create, update, patch, delete, deletecollection]
  # No resources listed = Matches everything else (including CRDs)
```

This will also match `wardle.example.com` resources — **no audit policy change needed**.

## Test Scenarios

All tests live under `Label("aggregated-api")` and are `Ordered`.

### Prerequisites (BeforeAll)

1. Verify sample-apiserver is running and APIService is Available
2. Verify `flunders.wardle.example.com` appears in `kubectl api-resources`
3. Create test namespace
4. Set up Gitea repo and credentials (same pattern as manager tests)
5. Create GitProvider and GitTarget

### S1 — API Discovery: aggregated resource is discoverable

**What it validates:** `FilterDiscoverableGVRs()` in
[internal/watch/discovery.go](../../internal/watch/discovery.go) accepts aggregated API resources.

**Action:**
1. Apply a WatchRule targeting `wardle.example.com/v1beta1/flunders`
2. Verify WatchRule reaches `Ready=True`

**Assert:**
- WatchRule status shows the resource was discovered and an informer was started
- Controller logs show the GVR was accepted by discovery filter

**Why it matters:** if the discovery filter silently drops aggregated API resources (e.g., due to
unexpected discovery response format), the entire pipeline is broken from the start.

---

### S2 — Audit: CREATE a Flunder produces a git commit

**What it validates:** the audit webhook receives events for aggregated API mutations and the full
pipeline (Redis → AuditConsumer → RuleStore match → sanitize → BranchWorker → git commit) works.

**Action:**
1. Create a Flunder resource in the test namespace via `kubectl create --as=e2e-user@test`
2. Wait for git commit

**Assert:**
- A file appears at `{target-path}/wardle.example.com/v1beta1/flunders/{namespace}/{name}.yaml`
  in the git repo
- The YAML content contains `apiVersion: wardle.example.com/v1beta1`, `kind: Flunder`
- The file contains the Flunder's `spec` fields
- The commit message attributes the author (from `--as`)

**Why it matters:** this is the core smoke test. Audit events for proxied API requests must arrive
with the same structure as native API requests.

---

### S3 — Audit: UPDATE a Flunder produces an updated git commit

**What it validates:** update mutations on aggregated resources flow through correctly.

**Action:**
1. Patch the Flunder's spec
2. Wait for git commit

**Assert:**
- The existing file is updated with the new spec
- The git log shows a new commit (not the same as the CREATE commit)

---

### S4 — Audit: DELETE a Flunder removes the file from git

**What it validates:** delete operations on aggregated resources.

**Action:**
1. Delete the Flunder
2. Wait for file removal in git

**Assert:**
- The file no longer exists in the repo
- A commit records the deletion

---

### S5 — Watch/Informer: snapshot reconcile captures existing Flunders

**What it validates:** the watch/informer system can list+watch aggregated API resources for
snapshot/reconcile, exercising the code in
[internal/watch/informers.go](../../internal/watch/informers.go).

**Action:**
1. Create two Flunder resources **before** the WatchRule exists
2. Apply the WatchRule
3. Wait for the initial snapshot sync (GitTarget condition `SnapshotSynced`)

**Assert:**
- Both pre-existing Flunders appear in git as part of the snapshot commit
- The commit is atomic (both resources in one commit, as per snapshot behavior)

**Why it matters:** the informer must successfully `list` aggregated API resources during its initial
cache sync. If the aggregation proxy introduces different response shapes or pagination behavior,
this test will catch it.

---

### S6 — Status subresource: status update behavior

**What it validates:** how the system handles status updates on aggregated API resources.

**Action:**
1. Create a Flunder
2. Wait for the initial CREATE commit
3. Update the Flunder's status subresource via
   `kubectl proxy` + `PUT /apis/wardle.example.com/v1beta1/namespaces/{ns}/flunders/{name}/status`

**Assert:**
- Check whether the audit policy filters or captures the status update
  - The existing audit policy drops `*/status` for core, apps, and networking groups — but does
    **not** drop `*/status` for arbitrary groups like `wardle.example.com`
  - Therefore, a status update should produce an audit event
- If an audit event arrives: verify the sanitizer handles it correctly (the sanitizer strips
  `status` by default for most resources via `preserveFields` only keeping `spec`, `data`,
  `binaryData`)
- Expected outcome: the status update either produces no git diff (because sanitize strips status)
  or is correctly filtered. **Either way, no spurious commit should appear.**

**Why it matters:** aggregated API servers commonly implement status subresources. The system must
handle these without producing spurious diffs or failing.

---

### S7 — Non-DNS-valid resource names in file paths

**What it validates:** the full pipeline handles resource instance names that are not valid DNS
labels.

**Background:** Kubernetes API aggregation allows extension servers to define their own validation.
The core API server enforces DNS label rules for its own resources, but proxied requests use the
extension server's validation. The `sample-apiserver` allows names that would be invalid as DNS
labels.

**Action:**
1. Create a Flunder with a name containing underscores: `test_flunder_with_underscores`
2. Create a Flunder with a name containing dots: `test.flunder.with.dots`
3. Wait for git commits

**Assert:**
- Files are created at the expected paths:
  - `{path}/wardle.example.com/v1beta1/flunders/{ns}/test_flunder_with_underscores.yaml`
  - `{path}/wardle.example.com/v1beta1/flunders/{ns}/test.flunder.with.dots.yaml`
- The git repository accepts these filenames (valid on all common filesystems)
- The content is correct
- Subsequent updates and deletes work for these names

**Why it matters:** [internal/types/identifier.go](../../internal/types/identifier.go) uses
`fmt.Sprintf` to build paths with no validation or escaping. This is the correct behavior — but
we should verify it works in practice. Characters like colons (`:`) are problematic on Windows
filesystems but fine on Linux/macOS and in bare git repos.

**Note on colons:** git itself handles colons in filenames, and our e2e cluster runs on Linux.
However, colons are invalid on Windows NTFS. If the git repo is cloned on Windows, files with
colons will fail. This is a documentation concern, not a code bug — the test should verify that
the Linux pipeline works and add a doc note about the Windows limitation.

---

### S8 — Cluster-scoped aggregated resource (Fischer)

**What it validates:** `ClusterWatchRule` works with cluster-scoped aggregated API resources.

**Action:**
1. Apply a ClusterWatchRule targeting `wardle.example.com/v1beta1/fischers` with scope `Cluster`
2. Create a Fischer resource
3. Wait for git commit

**Assert:**
- File appears at `{path}/wardle.example.com/v1beta1/fischers/{name}.yaml` (no namespace segment)
- ClusterWatchRule reaches `Ready=True`

---

### S9 — Extension server restart: watch recovery

**What it validates:** the informer reconnects after the aggregated API server restarts.

**Action:**
1. Have a WatchRule active for flunders with a Flunder already committed
2. Restart the sample-apiserver deployment: `kubectl rollout restart deployment/sample-apiserver`
3. Wait for the deployment to become Available again
4. Create a new Flunder
5. Wait for git commit

**Assert:**
- The new Flunder appears in git
- No errors in the gitops-reverser controller logs about the GVR becoming permanently unavailable

**Why it matters:** informer reconnection through the aggregation proxy is handled by the
Kubernetes client-go reflector, but we should verify it works in practice. The Watch Manager's
exponential backoff for unavailable GVRs
([internal/watch/manager.go](../../internal/watch/manager.go)) should handle this gracefully.

## Risks and Edge Cases to Document

### Audit event structure differences

The core API server generates audit events when proxying to an aggregated API server. The event
structure should be identical (same `objectRef`, same `responseObject` embedding), but this is an
assumption that needs validation. The test suite will confirm or disprove this.

Specific things to watch for:
- `objectRef.apiGroup` — should be `wardle.example.com`, not empty
- `objectRef.apiVersion` — should be `v1beta1`
- `objectRef.resource` — should be `flunders`
- `responseObject` — should contain the full Flunder object (depends on audit level)

### Discovery response format

`ServerPreferredResources()` should include aggregated API resources. The
[discovery code](../../internal/watch/discovery.go) parses `GroupVersion` strings and checks for
`list`/`watch` verbs. If the aggregation layer returns these differently, the discovery filter
would silently drop the resource. S1 validates this.

### Sanitizer behavior on unknown kinds

The [sanitizer](../../internal/sanitize/sanitize.go) preserves `spec`, `data`, `binaryData` and
selected top-level fields. For aggregated API resources with non-standard top-level fields (e.g.,
if a Flunder had a `config` field at the top level instead of under `spec`), those fields would be
lost. This is existing behavior and is the same for CRDs — but it's worth noting.

### Resource name character limits

While the `sample-apiserver` is permissive about names, there are filesystem limits to consider:
- Linux ext4/xfs: 255-byte filename limit (plenty for most resource names)
- Git: no inherent filename restrictions on Linux
- Windows NTFS: no `:`, `<`, `>`, `|`, `?`, `*` in filenames; 255-char path component limit

The test should document which characters are validated and note the Windows caveat.

### `sample-apiserver` version compatibility

The `sample-apiserver` tracks Kubernetes releases. The e2e cluster's k3s version determines which
`sample-apiserver` image tag to use. Pin the image tag and document the compatibility matrix:

| k3s version | sample-apiserver image tag |
|---|---|
| v1.31.x | `registry.k8s.io/e2e-test-images/sample-apiserver:1.31.1` |
| v1.32.x | `registry.k8s.io/e2e-test-images/sample-apiserver:1.32.1` |

Use a variable in the kustomization to make this easy to update.

## RBAC Changes Required

The gitops-reverser ClusterRole needs additional rules:

```yaml
- apiGroups: ["wardle.example.com"]
  resources: ["flunders", "fischers"]
  verbs: ["get", "list", "watch"]
```

This should be applied only when the aggregated-api test suite is running. Options:
1. **Conditional kustomize overlay** (recommended) — add a test-only RBAC patch in the
   sample-apiserver setup
2. A separate ClusterRole + ClusterRoleBinding that grants additional permissions to the
   gitops-reverser service account

Option 1 is cleaner. The `_sample-apiserver-ready` task applies the overlay.

## Open Questions

1. **Should S5 (snapshot) be a separate test or combined with S2?** — Keeping them separate is
   cleaner. S2 tests the live audit path; S5 tests the watch/informer snapshot path. They exercise
   different code.

2. **Should non-DNS-valid names include colons?** — Colons work on Linux but break on Windows.
   Testing colons validates our Linux pipeline but may set an expectation we can't keep for
   Windows users. Recommendation: test underscores and dots (universally safe), document the
   colon/Windows limitation.

3. **Do we need to test the `sample-apiserver` with in-memory storage, or etcd-backed?** —
   In-memory (`--etcd-servers=""`) is sufficient. Data doesn't need to survive pod restarts for
   e2e tests, and it avoids deploying another etcd instance. S9 (restart recovery) tests our
   operator's resilience, not the extension server's.

4. **Integration with the full test suite** — the `aggregated-api` tests require the
   sample-apiserver infrastructure. Running `test-e2e-full` should either:
   (a) skip aggregated-api tests if sample-apiserver is not deployed, or
   (b) always deploy sample-apiserver as part of `prepare-e2e`.
   Recommendation: (a) for now — use a Ginkgo `BeforeAll` check that skips the suite if
   `kubectl api-resources | grep flunders` fails. Later, promote to (b) if the tests prove stable.
