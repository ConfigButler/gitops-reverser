# E2E Test Design: Aggregated API Server (Extension API Server)

> Last updated: April 2026

## Motivation

GitOps Reverser currently has e2e coverage for built-in resources (ConfigMaps, Secrets, Deployments,
etc.) and CRD-backed resources (IceCreamOrder). However, Kubernetes supports a third category of
resources: those served by **aggregated API servers** (extension API servers), registered via
`APIService` objects.

These resources participate in the Kubernetes API surface differently from CRDs:

- They are served by a separate process (not etcd-backed through the core API server)
- The core API server proxies requests to the extension server
- Audit events are generated by the core API server's proxy layer, not the extension server
- Watch streams are proxied through the aggregation layer
- Status subresources may behave differently depending on the extension server implementation

We need to verify that the full gitops-reverser pipeline — audit ingestion, watch/informer
reconcile, sanitization, file path generation, and git commit — works correctly with aggregated API
resources.

### What the existing docs already say

[kubernetes-api-discovery.md](kubernetes-api-discovery.md) already states that aggregated API
resources are theoretically supported: "yes, you can watch resources served through `APIService`;
CRD is not required." However, this has **never been tested end-to-end**. This design closes that
gap.

The concrete bootstrap and wiring plan for the proxy-backed environment lives in
[e2e-aggregated-apiserver-proxy-hookup-plan.md](e2e-aggregated-apiserver-proxy-hookup-plan.md).
This document focuses on **what the tests must prove** once that environment exists.

### What we need to prove about gitops-reverser vs. what is Kubernetes behavior

This document is careful to separate concerns:

- **Gitops-reverser behavior** (our code): audit ingestion, rule matching, sanitization, file path
  generation, watch/informer reconcile, content dedup, git commit. These are what the tests must
  assert on.
- **Kubernetes aggregation mechanics** (not our code): audit event generation for proxied requests,
  watch stream proxying, APIService health checks. We depend on these but do not test them directly.
  The tests will *exercise* them but should not assert on Kubernetes internals.

---

## Scope

This test suite validates:

1. **Audit webhook behavior** — do audit events for aggregated API resources arrive with the same
   structure (verb, objectRef, responseObject) as CRD/built-in resources?
2. **Watch/informer behavior** — can informers list+watch aggregated API resources for
   snapshot/reconcile?
3. **Status subresource handling** — does a status update on an aggregated resource get correctly
   deduplicated (no spurious commit)?
4. **Non-standard resource names** — do resource instance names with underscores and dots (permitted
   by the sample-apiserver's path-segment validation) survive the full pipeline?
5. **API discovery integration** — does `FilterDiscoverableGVRs()` correctly discover and accept
   aggregated API resources?
6. **Sanitizer field preservation** — does the sanitizer correctly handle aggregated API resources,
   including the known limitation for non-standard top-level fields?
7. **Reconcile/snapshot** — does an initial snapshot correctly capture existing aggregated API
   resources?

Out of scope:
- Extension API server high availability or failure modes
- Performance comparison between CRD and aggregated API watches
- mTLS between the core API server and extension server (handled by Kubernetes, not our code)

## Test Infrastructure: Which Extension API Server to Use

### Decision: `kubernetes/sample-apiserver`

The Kubernetes project maintains
[sample-apiserver](https://github.com/kubernetes/sample-apiserver) — a minimal extension API
server that serves resources in the `wardle.example.com` API group. This is the canonical example
used by Kubernetes itself for testing aggregated APIs.

**Resources it serves:**

| Version | Resource | Kind | Scope | Notes |
|---|---|---|---|---|
| `v1alpha1` | `flunders` | Flunder | Namespaced | Has `spec` and `status` fields |
| `v1alpha1` | `fischers` | Fischer | Cluster | Has `disallowedFlunders` as a **top-level** field (not under `spec`) |
| `v1beta1` | `flunders` | Flunder | Namespaced | `spec` uses `flunderReference`/`fischerReference` instead of v1alpha1's `reference` |

Note: `fischers` is only served at `v1alpha1`, not `v1beta1`.

**Why this is the right choice:**
- Official Kubernetes project, actively maintained alongside k8s releases
- Minimal footprint: single binary, simple schema
- Has both namespaced and cluster-scoped resources
- Has a `status` subresource on Flunder
- Supports all CRUD verbs plus `list` and `watch`
- Used in Kubernetes' own e2e tests
- Pre-built container images at `registry.k8s.io/e2e-test-images/sample-apiserver`
- **The upstream `artifacts/example/` directory provides battle-tested deployment manifests**
  including all required RBAC

**Alternatives considered and rejected:**
- **metrics-server**: read-only (no create/update/delete), unsuitable for mutating audit tests
- **Custom extension server**: duplicates sample-apiserver's purpose, adds maintenance burden
- **HTTP mock behind APIService**: too fragile, must implement full K8s API contract

### Name validation behavior (verified from source)

The sample-apiserver uses `genericregistry.Store` which defaults to `path.ValidatePathSegmentName`
for instance name validation. This rejects only `/`, `%`, and `..` — it **does allow** underscores,
dots, and other characters that would be invalid in DNS labels. This is confirmed by reading the
[flunder strategy](../../external-sources/sample-apiserver/pkg/registry/wardle/flunder/strategy.go)
and [REST store](../../external-sources/sample-apiserver/pkg/registry/wardle/flunder/etcd.go).

This means S7 (non-standard names with underscores and dots) is viable without modifying the
sample-apiserver.

## Setup: Providing `sample-apiserver` in the e2e environment

The concrete installation mechanism is intentionally separated from this test-design document.

- This document defines the required environment capabilities and the assertions the tests should
  make.
- The active bootstrap ownership and rollout sequence are defined in
  [e2e-aggregated-apiserver-proxy-hookup-plan.md](e2e-aggregated-apiserver-proxy-hookup-plan.md).

At the time of writing, the preferred direction is to have Flux bootstrap the aggregated API stack
through the `apiservice-audit-proxy` chart rather than keep a permanently hand-maintained parallel
manifest bundle in this repo.

### TLS approach: `insecureSkipTLSVerify`

The upstream `artifacts/example/apiservice.yaml` uses `insecureSkipTLSVerify: true` on the
APIService. This skips TLS verification between the core API server and the extension server.

**We use this approach for e2e.** It avoids:
- Creating a cert-manager Certificate (the repo has a namespaced
  `gitops-reverser-selfsigned-issuer` Issuer, not a ClusterIssuer, and cert-manager's
  `cainjector` annotation wiring for APIService caBundle injection is fiddly)
- Injecting CA bundles into the APIService
- Debugging cert-related startup failures

The sample-apiserver generates its own self-signed serving cert at startup when no cert files are
provided. Combined with `insecureSkipTLSVerify`, this just works.

This matches how the sample-apiserver is deployed in Kubernetes' own test infrastructure.

### Deployment artifacts

Add the manifests to the existing `test/e2e/setup/manifests/` directory, following the same
pattern used for Prometheus (plain manifests applied via kustomize). This keeps all non-Helm
e2e infrastructure in one place:

```
test/e2e/setup/manifests/
├── kustomization.yaml            # add sample-apiserver/ resources here
├── prometheus/                   # existing
└── sample-apiserver/
    ├── namespace.yaml            # wardle namespace
    ├── sa.yaml                   # ServiceAccount: apiserver
    ├── deployment.yaml           # wardle-server + etcd sidecar
    ├── service.yaml              # ClusterIP service on port 443
    ├── apiservice.yaml           # v1alpha1.wardle.example.com with insecureSkipTLSVerify
    ├── auth-delegator.yaml       # ClusterRoleBinding → system:auth-delegator
    ├── auth-reader.yaml          # RoleBinding → extension-apiserver-authentication-reader
    ├── rbac.yaml                 # ClusterRole for namespace/webhook/flowcontrol access
    └── rbac-bind.yaml            # ClusterRoleBinding for above
```

The existing `test/e2e/setup/manifests/kustomization.yaml` gets the new resources added:

```yaml
resources:
  - prometheus/instance.yaml
  - prometheus/prometheus-rbac.yaml
  - prometheus/servicemonitor.yaml
  - sample-apiserver/namespace.yaml
  - sample-apiserver/sa.yaml
  - sample-apiserver/deployment.yaml
  - sample-apiserver/service.yaml
  - sample-apiserver/apiservice.yaml
  - sample-apiserver/auth-delegator.yaml
  - sample-apiserver/auth-reader.yaml
  - sample-apiserver/rbac.yaml
  - sample-apiserver/rbac-bind.yaml
```

This means the sample-apiserver manifests are installed automatically as part of `_services-ready`
(which already runs `kubectl apply -k test/e2e/setup/manifests`). There is no separate
"apply sample-apiserver" step in the Taskfile; we only add a readiness-wait task afterwards.

### RBAC: full set from upstream (all four resources required)

The upstream example defines four RBAC resources, and **all four are required** for the
sample-apiserver to function:

1. **`auth-delegator.yaml`** — ClusterRoleBinding granting `system:auth-delegator` to the
   ServiceAccount. Required for the extension server to delegate authentication decisions to
   the core API server.

2. **`auth-reader.yaml`** — RoleBinding in `kube-system` granting
   `extension-apiserver-authentication-reader` to the ServiceAccount. **This is the one most often
   missed.** Without it, the extension server cannot read the `extension-apiserver-authentication`
   ConfigMap in `kube-system`, and delegated auth fails even when the pod is healthy and the
   APIService exists.

3. **`rbac.yaml`** — ClusterRole granting access to namespaces, admission webhook configs, and
   flow control resources. Required for the extension server's internal machinery.

4. **`rbac-bind.yaml`** — ClusterRoleBinding for the above ClusterRole.

### Key resources

**APIService** (the simple approach):
```yaml
apiVersion: apiregistration.k8s.io/v1
kind: APIService
metadata:
  name: v1alpha1.wardle.example.com
spec:
  insecureSkipTLSVerify: true
  group: wardle.example.com
  groupPriorityMinimum: 1000
  versionPriority: 15
  service:
    name: api
    namespace: wardle
  version: v1alpha1
```

**Deployment** (with etcd sidecar for storage):
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: wardle-server
  namespace: wardle
spec:
  replicas: 1
  selector:
    matchLabels:
      apiserver: "true"
  template:
    metadata:
      labels:
        apiserver: "true"
    spec:
      serviceAccountName: apiserver
      containers:
      - name: wardle-server
        image: registry.k8s.io/e2e-test-images/sample-apiserver:TAG
        args: ["--etcd-servers=http://localhost:2379"]
      - name: etcd
        image: registry.k8s.io/etcd:v3.6.8
```

The etcd sidecar is required because the sample-apiserver's `--etcd-servers` flag does not accept
an empty value — it needs a real etcd. The sidecar pattern from the upstream example is the
simplest approach. Data does not need to survive pod restarts for e2e.

### Image tag: must match k3s version

The e2e cluster uses `rancher/k3s:v1.35.2-k3s1`. The sample-apiserver image must match the
Kubernetes minor version. Use a kustomize variable for easy updates:

```yaml
# kustomization.yaml
images:
  - name: registry.k8s.io/e2e-test-images/sample-apiserver
    newTag: "1.35.1"   # match k3s v1.35.x
```

**Action required before implementation:** verify that `registry.k8s.io/e2e-test-images/sample-apiserver:1.35.1`
exists. If not, check the available tags and use the closest match. The sample-apiserver images
are published per Kubernetes release; newer k3s versions may require building from source.

### gitops-reverser RBAC: no changes needed

The existing ClusterRole at `config/rbac/role.yaml` already grants wildcard read access:

```yaml
- apiGroups: ['*']
  resources: ['*']
  verbs: [get, list, watch]
```

This covers `wardle.example.com` flunders and fischers. No additional RBAC overlay is needed for
the gitops-reverser controller.

### Integration into `prepare-e2e`

The sample-apiserver is deployed as part of the standard `prepare-e2e` flow, not as an opt-in.
This keeps the provisioning story simple and avoids runtime skip-or-fail ambiguity in the test
suite.

Since the manifests are now part of `test/e2e/setup/manifests/`, they are applied by the existing
`_services-ready` task. We only need a readiness-wait task that depends on `_services-ready`:

```yaml
_sample-apiserver-ready:
  method: timestamp
  sources:
    - '{{.CS}}/services.ready'
    - test/e2e/setup/manifests/sample-apiserver/**/*
  generates:
    - '{{.CS}}/sample-apiserver.ready'
  status:
    - test -f "{{.CS}}/sample-apiserver.ready"
  cmds:
    - task: _services-ready
    - |
      # Wait for the deployment to be available
      kubectl --context "{{.CTX}}" -n wardle wait deployment/wardle-server \
        --for=condition=Available --timeout=120s
      # Wait for APIService to become available (core API server can reach extension server)
      kubectl --context "{{.CTX}}" wait apiservice/v1alpha1.wardle.example.com \
        --for=condition=Available --timeout=120s
      # Verify the resource is discoverable
      kubectl --context "{{.CTX}}" api-resources --api-group=wardle.example.com | grep -q flunders
      touch "{{.CS}}/sample-apiserver.ready"
```

Add `_sample-apiserver-ready` to the `prepare-e2e` flow before `_prepare-e2e-ready`, and make
`_prepare-e2e-ready` consume the resulting stamp:
```yaml
prepare-e2e:
  cmds:
    # ... existing steps ...
    - task: _webhook-tls-ready
    - task: _sample-apiserver-ready   # NEW
    - task: _prepare-e2e-ready
    - task: portforward-ensure

_prepare-e2e-ready:
  method: timestamp
  sources:
    # ... existing sources ...
    - '{{.CS}}/sample-apiserver.ready'     # NEW
  # ...
```

And add a standalone task for running just the aggregated-api tests:
```yaml
test-e2e-aggregated-api:
  desc: Run aggregated API server e2e scenarios
  cmds:
    - |
      export CTX="{{.CTX}}"
      export INSTALL_MODE="{{.INSTALL_MODE}}"
      export NAMESPACE="{{.NAMESPACE}}"
      export E2E_AGE_KEY_FILE="{{.CS}}/age-key.txt"
      go test -timeout "{{.E2E_GO_TEST_TIMEOUT}}" ./test/e2e/ -v -ginkgo.v \
        -ginkgo.label-filter=aggregated-api
```

### Audit policy

The existing audit policy at `test/e2e/cluster/audit/policy.yaml` has a catch-all rule:

```yaml
- level: RequestResponse
  omitManagedFields: true
  verbs: [create, update, patch, delete, deletecollection]
  # No resources listed = Matches everything else (including CRDs)
```

This matches `wardle.example.com` resources. **No audit policy change needed.**

However, note that the existing noise filter drops `*/status` for `""`, `apps`, and
`networking.k8s.io` groups — but does **not** drop `*/status` for `wardle.example.com`. This
means status-subresource updates on Flunders **will** produce audit events. This is relevant for
S6.

## Test Scenarios

All tests live under `Label("aggregated-api")` and are `Ordered`.

Recommended label split:

- `S2` should also carry `Label("smoke")`
- the remaining scenarios stay `aggregated-api` only until we have runtime data showing the
  always-on infrastructure cost is negligible

### Prerequisites (BeforeAll)

1. Verify `flunders.wardle.example.com` appears in `kubectl api-resources` — if not, fail fast
   with a clear error ("sample-apiserver not deployed; run task prepare-e2e")
2. Create test namespace
3. Set up Gitea repo and credentials (same pattern as manager tests)
4. Create GitProvider and GitTarget

### S1 — Discovery + WatchRule: aggregated resource is accepted

**What it validates:** the combination of `FilterDiscoverableGVRs()` in
[internal/watch/discovery.go](../../internal/watch/discovery.go) and the WatchRule controller
correctly accepting an aggregated API resource.

**Action:**
1. Apply a WatchRule targeting `wardle.example.com/v1alpha1/flunders`
2. Wait for WatchRule to reach `Ready=True`

**Assert:**
- `verifyResourceStatus("watchrule", name, ns, "True", "Ready", "")` — the WatchRule is Ready

This is the only status signal the WatchRule currently exposes. We do **not** assert on informer
startup or discovery internals — `Ready=True` is the contract. If the discovery filter silently
dropped the GVR, the WatchRule would not reach Ready.

**Why it matters:** if `ServerPreferredResources()` returns a different response shape for
aggregated APIs, or the discovery filter has an edge case, this catches it at the entry point.

---

### S2 — Audit: CREATE a Flunder produces a git commit

**What it validates:** the full audit pipeline for aggregated API mutations:
Redis → AuditConsumer → RuleStore match → sanitize → BranchWorker → git commit.

**Labels:** `aggregated-api`, `smoke`

**Action:**
1. Create a Flunder resource in the test namespace via `kubectl create --as=e2e-user@test`
2. Wait for git commit

**Assert:**
- A file appears at `{target-path}/wardle.example.com/v1alpha1/flunders/{namespace}/{name}.yaml`
- The YAML content contains `apiVersion: wardle.example.com/v1alpha1`, `kind: Flunder`
- The file contains the Flunder's `spec` fields
- The commit message attributes the author (from `--as`)

**Why it matters:** this is the core smoke test. If audit events for proxied API requests have a
different structure than native requests, the pipeline would silently drop them.

This is the best candidate for smoke coverage because it proves the end-to-end aggregated API happy
path with a single resource create, without pulling in the more brittle edge-case behaviors from the
rest of the suite.

### S3 — Audit: UPDATE a Flunder produces an updated git commit

**What it validates:** update mutations on aggregated resources flow through correctly.

**Action:**
1. Patch the Flunder's spec (e.g., change `referenceType`)
2. Wait for git commit

**Assert:**
- The file is updated with the new spec

### S4 — Audit: DELETE a Flunder removes the file from git

**What it validates:** delete operations on aggregated resources.

**Action:**
1. Delete the Flunder
2. Wait for file removal in git

**Assert:**
- The file no longer exists in the repo
- A commit records the deletion

### S5 — Watch/Informer: snapshot reconcile captures existing Flunders

**What it validates:** informers can list+watch aggregated API resources for snapshot/reconcile,
exercising [internal/watch/informers.go](../../internal/watch/informers.go).

**Action:**
1. Create two Flunder resources **before** the WatchRule exists
2. Apply the WatchRule
3. Wait for the initial snapshot sync (GitTarget condition `SnapshotSynced`)

**Assert:**
- Both pre-existing Flunders appear in git as part of the snapshot commit
- The commit is atomic (both resources in one commit, as per snapshot behavior)

**Why it matters:** the informer must successfully `list` aggregated API resources during initial
cache sync. If the aggregation proxy introduces different pagination or response shapes, this
catches it.

---

### S6 — Status subresource: no spurious commit

**What it validates:** a status-only update on a Flunder does not produce a spurious git commit.

**Expected behavior (concrete):** the sanitizer strips `status` because it only preserves `spec`,
`data`, and `binaryData` (plus the hardcoded top-level fields list in
[sanitize.go:74](../../internal/sanitize/sanitize.go#L74)). After sanitization, the content hash
is identical to the previous version. The content-hash deduplication in
[GitTargetEventStream](../../internal/reconcile/git_target_event_stream.go) drops it. **No new
commit should appear.**

**Action:**
1. Create a Flunder, wait for the CREATE commit, record the git HEAD SHA
2. Update the Flunder's status subresource:
   ```
   kubectl --context $CTX patch flunder $NAME -n $NS --subresource=status \
     --type=merge -p '{"status":{}}'
   ```
   (Note: FlunderStatus is an empty struct, but the patch still triggers an audit event)
3. Wait a reasonable interval (e.g., 15 seconds — longer than the push interval)
4. Check git HEAD SHA

**Assert:**
- Git HEAD SHA is unchanged — no new commit was created
- No error in controller logs related to flunder status processing

**Why it matters:** the audit policy does not filter `*/status` for `wardle.example.com`. Without
sanitization + dedup working correctly, every status update would create a spurious commit.

---

### S7 — Non-standard resource instance names

**What it validates:** the full pipeline handles resource instance names that contain underscores
and dots — characters that are valid per the sample-apiserver's path-segment validation but would
be invalid as DNS labels.

**Confirmed:** the sample-apiserver uses `path.ValidatePathSegmentName` (via
`genericregistry.Store` defaults), which allows underscores and dots. Only `/`, `%`, and `..` are
rejected.

**Action:**
1. Create a Flunder named `test_flunder_underscore`
2. Create a Flunder named `test.flunder.dotted`
3. Wait for git commits

**Assert:**
- Files are created at:
  - `{path}/wardle.example.com/v1alpha1/flunders/{ns}/test_flunder_underscore.yaml`
  - `{path}/wardle.example.com/v1alpha1/flunders/{ns}/test.flunder.dotted.yaml`
- Content is correct YAML with the expected apiVersion/kind
- Subsequent update and delete work for both names

**Why it matters:** [internal/types/identifier.go](../../internal/types/identifier.go) uses
`fmt.Sprintf` to build paths with no validation or escaping. Underscores and dots are safe on all
common filesystems and in git. This test confirms the pipeline handles them end-to-end.

---

### S8 — Cluster-scoped aggregated resource (Fischer) + sanitizer edge case

**What it validates:** `ClusterWatchRule` works with cluster-scoped aggregated API resources.
Also surfaces a **known sanitizer limitation**.

**Action:**
1. Apply a ClusterWatchRule targeting `wardle.example.com/v1alpha1/fischers` with scope `Cluster`
2. Create a Fischer resource with `disallowedFlunders: ["bad-flunder"]`
3. Wait for git commit

**Assert:**
- File appears at `{path}/wardle.example.com/v1alpha1/fischers/{name}.yaml` (no namespace segment)
- ClusterWatchRule reaches `Ready=True`

**Known issue to document:** Fischer has `disallowedFlunders` as a top-level field, not under
`spec`. The sanitizer
([sanitize.go:37](../../internal/sanitize/sanitize.go#L37)) only preserves `spec`, `data`,
`binaryData`, and a hardcoded list of other top-level fields. `disallowedFlunders` is **not** in
that list, so it will be silently dropped from the git output.

This is the same behavior for any CRD or aggregated API resource that uses non-standard top-level
fields. The test should:
1. Assert that the file is created (the resource is tracked)
2. Assert that `disallowedFlunders` is **absent** from the committed YAML (documenting current
   behavior)
3. Add a code comment noting this is a known limitation of the sanitizer, not a bug in aggregated
   API support

---

### S9 — Extension server restart: watch recovery

**What it validates:** the informer reconnects after the aggregated API server restarts.

**Action:**
1. Have a WatchRule active for flunders with a Flunder already committed
2. Restart the sample-apiserver: `kubectl -n wardle rollout restart deployment/wardle-server`
3. Wait for the deployment to become Available again
4. Wait for the APIService to become Available again
5. Create a new Flunder
6. Wait for git commit

**Assert:**
- The new Flunder appears in git
- No persistent errors in the gitops-reverser controller logs about the GVR becoming unavailable

**Why it matters:** informer reconnection through the aggregation proxy is handled by client-go's
reflector. The Watch Manager's exponential backoff for unavailable GVRs
([internal/watch/manager.go](../../internal/watch/manager.go)) should handle the transient
unavailability gracefully.

## Risks and Edge Cases

### Audit event structure for proxied requests

The core API server generates audit events when proxying to an aggregated API server. The event
structure should be identical (same `objectRef`, same `responseObject` embedding). S2 will confirm
or disprove this assumption.

Specific things to watch for:
- `objectRef.apiGroup` — should be `wardle.example.com`, not empty
- `objectRef.apiVersion` — should be `v1alpha1`
- `objectRef.resource` — should be `flunders`
- `responseObject` — should contain the full Flunder object

### Sanitizer field-loss for non-standard schemas

As detailed in S8, any aggregated API resource (or CRD) that places data in top-level fields
outside the hardcoded preservation list will lose that data. Fischer's `disallowedFlunders` is the
concrete example. This is a pre-existing product limitation — not introduced by aggregated API
support — but the test suite should document it clearly.

### etcd sidecar adds resource overhead

The sample-apiserver deployment includes an etcd sidecar. This adds ~100MB memory to the test
cluster. Acceptable for e2e, but worth noting for resource-constrained CI runners.

### Version pinning

The sample-apiserver image must match the k3s Kubernetes version. Currently k3s `v1.35.2-k3s1`.
The kustomize image tag makes this a one-line change when k3s is upgraded.

```yaml
apiVersion: configbutler.ai/v1alpha1
kind: WatchRule
metadata:
  name: {{ .Name }}
  namespace: {{ .Namespace }}

  targetRef:
    name: {{ .GitTargetName }}
```

```yaml
apiVersion: wardle.example.com/v1alpha1
```

```yaml
apiVersion: wardle.example.com/v1alpha1

  name: {{ .Name }}
```
