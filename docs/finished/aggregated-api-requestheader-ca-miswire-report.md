# Aggregated API Proxy RequestHeader CA Miswire Report

## Summary

During the April 24, 2026 e2e hookup of the Flux-bootstrapped aggregated API stack, the proxy was
briefly configured to verify the inbound kube-apiserver/front-proxy client certificate against the
**backend mTLS CA**:

- wrong trust root: `audit-pass-through-backend-client-ca`
- correct trust root for delegated identity: the cluster's
  `extension-apiserver-authentication` requestheader client CA

That configuration was wrong because those are two different TLS relationships:

1. `kube-apiserver -> apiservice-audit-proxy`
2. `apiservice-audit-proxy -> sample-apiserver`

The fix used in this repo for the first implementation pass was simple:

- remove `requestHeader.clientCASecretName` from
  `test/e2e/setup/flux/values/aggregated-api-values.yaml`
- keep the backend client certificate wiring in place

That restored a working aggregated API bootstrap for e2e.

## Observed Failure

The proxy pod started, but the `APIService` stayed unavailable. The relevant symptoms were:

- `kubectl get apiservice v1alpha1.wardle.example.com` showed `Available=False`
- aggregated discovery did not surface `flunders`
- proxy logs showed repeated TLS handshake failures from the apiserver

Observed failure message:

```text
remote error: tls: unknown certificate authority
```

Observed `APIService` status:

```text
failing or missing response from https://10.42.0.8:9445/apis/wardle.example.com/v1alpha1
```

## Root Cause

The misconfiguration came from treating this chart value:

```yaml
requestHeader:
  clientCASecretName: ...
```

as if it should reuse the same CA that the proxy uses when authenticating **to the backend**.

That is incorrect.

`requestHeader.clientCASecretName` configures the proxy's `--client-ca-file`, which is used to
verify the certificate presented by the **immediate inbound client** before trusting delegated
`X-Remote-*` headers. In an aggregated API request, that inbound client is kube-apiserver's
front-proxy identity, not the sample-apiserver backend.

The backend CA generated by the chart is for a different purpose:

- the sample-apiserver trusts the proxy client cert signed by
  `audit-pass-through-backend-client-ca`

It says nothing about whether the proxy should trust kube-apiserver's front-proxy client cert.

## Why The E2E Fix Was Safe

For this repo's current e2e goal, we only needed:

- the proxy to serve the aggregated API successfully
- the proxy to emit rich audit-backed events
- the e2e suite to prove aggregated API behavior end to end

We did **not** need to solve production-grade delegated-header trust in this pass.

So the safe short-term e2e choice was:

- do not set `--client-ca-file`
- allow the proxy to accept the inbound aggregated request without front-proxy cert validation

This matches the non-goal already called out in the aggregated API hookup plan:

- do not solve production-grade delegated-header trust in this pass

## Current Repo State

The repo now intentionally omits requestheader CA wiring in:

- `test/e2e/setup/flux/values/aggregated-api-values.yaml`

The repo still keeps the backend mTLS pieces:

- `backend.clientCertSecretName`
- `testApiserver.backendClientCert.*`

That gives the working relationship:

- `apiservice-audit-proxy -> sample-apiserver` is authenticated

while intentionally leaving this out for now:

- `kube-apiserver/front-proxy -> apiservice-audit-proxy` requestheader client cert verification

## Does This Need An Upstream Fix?

Yes, but mostly as a **chart usability and regression-prevention** fix, not as a core proxy-code
bug fix.

### What does not seem broken upstream

The upstream proxy behavior itself is correct:

- `--client-ca-file` means "verify the inbound front-proxy client"
- backend client certificate settings are separate

The upstream repo already contains the right conceptual model and even a helper script for writing
the real requestheader CA from the cluster:

- `external-sources/apiservice-audit-proxy/hack/e2e/write-requestheader-client-ca.sh`

### What is still too easy to get wrong

The chart is easy to miswire if an integrator assumes:

- "there is already a CA for proxy/backend auth, so I can reuse that for requestheader auth"

That is exactly the mistake made during this hookup.

So the upstream improvement should focus on making this confusion harder:

1. Clarify chart docs and values comments that `requestHeader.clientCASecretName` is for the
   **kube-apiserver/front-proxy CA**, not the backend client CA.
2. Keep demo/example values from encouraging reuse of backend TLS assets for requestheader trust.
3. Add an upstream e2e/assertion path that proves requestheader verification is wired with the real
   `extension-apiserver-authentication` CA when enabled.

## Does This Need An Upstream Test?

Yes. A regression test is worthwhile.

Recommended upstream coverage:

1. Install the chart with `requestHeader.clientCASecretName` set from the real
   `extension-apiserver-authentication` config.
2. Verify the `APIService` becomes `Available=True`.
3. Make an impersonated aggregated API request.
4. Assert the proxy accepts the request and preserves delegated user identity.

Useful negative coverage:

1. Install with `requestHeader.clientCASecretName` pointing at the backend client CA.
2. Verify the `APIService` does **not** become healthy, or at minimum that the proxy logs the TLS
   verification failure.

That negative case would directly catch the miswire hit in this repo.

## Recommended Upstream Follow-Up

The best upstream follow-up is probably:

1. docs/comment hardening in the chart values around `requestHeader.*`
2. one e2e path that enables requestheader verification correctly
3. optionally one negative test or fixture proving backend-client CA reuse is wrong

I would treat the **test** as the important part. The chart feature already works when given the
right secret; what was missing was a guardrail against a very plausible integration mistake.
