Install GitOps Reverser first. After that, generate the kube-apiserver audit webhook kubeconfig, copy it to every
control-plane node, configure kube-apiserver, and restart or reload kube-apiserver once.

Recommended kube-apiserver settings:
- use `--audit-webhook-mode=batch`
- do NOT start with `blocking` or `blocking-strict`

Audit-pipeline readiness:
- The pod reports NotReady on `/readyz` until the audit ingress listener is accepting{{ if .Values.servers.audit.tls.enabled }} with a loaded TLS cert{{ end }} and it has made its first connection to Redis/Valkey.
- Because kube-apiserver reaches the controller through the audit Service (`{{ $auditServiceName }}`), a NotReady pod is removed from the Service endpoints. With `--audit-webhook-mode=batch` the apiserver then buffers and retries during a rollout or cert rotation instead of dropping the first events into a pod that cannot yet receive them.
- This endpoint gating only helps when the apiserver targets the Service. If you point `--audit-webhook-config-file` at a node URL or an external load balancer that bypasses endpoint health, readiness does NOT gate routing — rely on `batch` mode + the apiserver retry buffer.
- Liveness (`/healthz`) stays a bare process check, so a Redis or cert outage never crashloops the pod.

- `ClusterIP` alone is usually NOT enough unless your control-plane host can route to the service CIDR.

Copy-paste example to generate `audit-webhook.kubeconfig` after the Secrets exist:

```bash
export NAMESPACE={{ .Release.Namespace }}
export AUDIT_ROOT_CA_SECRET={{ $auditRootCASecret }}
export AUDIT_CLIENT_SECRET={{ $auditClientSecret }}
export AUDIT_TLS_SERVER_NAME={{ $tlsServerName }}
export AUDIT_WEBHOOK_SERVER_URL={{- if and (eq .Values.auditService.type "NodePort") $auditNodePort }}https://127.0.0.1:{{ $auditNodePort }}/audit-webhook{{- else if $auditClusterIP }}https://{{ $auditClusterIP }}:{{ $auditServerPort }}/audit-webhook{{- else }}https://<reachable-address>/audit-webhook{{- end }}

cat > audit-webhook.kubeconfig <<EOF
apiVersion: v1
kind: Config
preferences: {}
clusters:
- name: audit-webhook
  cluster:
    server: ${AUDIT_WEBHOOK_SERVER_URL}
{{- if .Values.auditKubeconfig.insecureSkipTLSVerify }}
    insecure-skip-tls-verify: true
{{- else }}
    certificate-authority-data: $(kubectl get secret "${AUDIT_ROOT_CA_SECRET}" -n "${NAMESPACE}" -o jsonpath='{.data.tls\.crt}')
    tls-server-name: ${AUDIT_TLS_SERVER_NAME}
{{- end }}
contexts:
- name: webhook
  context:
    cluster: audit-webhook
    user: kube-apiserver
current-context: webhook
users:
- name: kube-apiserver
  user:
    client-certificate-data: $(kubectl get secret "${AUDIT_CLIENT_SECRET}" -n "${NAMESPACE}" -o jsonpath='{.data.tls\.crt}')
    client-key-data: $(kubectl get secret "${AUDIT_CLIENT_SECRET}" -n "${NAMESPACE}" -o jsonpath='{.data.tls\.key}')
EOF
```
{{- if .Values.auditKubeconfig.insecureSkipTLSVerify }}

`auditKubeconfig.insecureSkipTLSVerify=true` is enabled, so the generated kubeconfig skips audit server certificate
validation. This is for local/bootstrap scenarios only, not a production recommendation.
{{- end }}
{{- else }}
`servers.audit.tls.enabled=false`, so only an insecure development bootstrap makes sense:

```yaml
server: https://<reachable-address>/audit-webhook
insecure-skip-tls-verify: true
```

{{- end }}

After generating the kubeconfig:
1. Copy `audit-webhook.kubeconfig` to every control-plane node.
2. Create or update your audit policy file.
3. Start kube-apiserver with:
   - `--audit-policy-file=/path/to/audit-policy.yaml`
   - `--audit-webhook-config-file=/path/to/audit-webhook.kubeconfig`
   - `--audit-webhook-mode=batch`
4. Restart or reload kube-apiserver once.
